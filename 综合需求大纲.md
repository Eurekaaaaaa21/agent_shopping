# 电商平台 + 多智能体客服 —— 综合需求大纲

---

## 一、项目概述

### 1.1 项目组成

本项目由**两个子系统**构成，通过内部 HTTP 接口协作，共同组成一个完整的电商平台：

| 子系统 | 定位 | 说明 |
|--------|------|------|
| **shop-service** | 电商基础服务 | 承载用户、商品、购物车、订单、支付、物流、售后的完整交易链路 |
| **ai-service** | AI 智能客服 | 多智能体协同系统，提供 FAQ 检索 + 实时业务数据查询 |

### 1.2 协作关系

```
用户浏览器 / 前端
       │
       ▼
   Nginx (统一入口)
       │
       ├──── /api/shop/*  ──►  shop-service（电商业务）
       │
       └──── /api/ai/*    ──►  ai-service（AI 客服）
                                  │
                                  │ 内部 HTTP 调用（透传用户 JWT）
                                  ▼
                              shop-service 内部接口
```

- 用户在 C 端页面与 AI 客服对话，AI 客服通过调用电商**内部接口**获取实时数据（订单/物流/售后/商品），生成自然语言回答
- AI 客服**不直连数据库**，所有业务数据通过 shop-service 暴露的内部接口获取
- 内部接口**透传用户 JWT**：ai-service 将前端用户请求中的 JWT 原样传递给 shop-service，shop-service 自行解析 JWT 获取 user_id 并校验权限，不信任调用方传参

### 1.3 核心闭环链路（总览）

```
注册 → 登录 → 浏览商品 → 加入购物车 → 下单(锁库存) → 模拟支付
    → 物流追踪(模拟数据) → 签收 → 售后申请 → 管理员审核 → 完成

管理员：分类管理 → 商品发布/上下架 → 缓存更新 → C 端可见

AI 客服：用户提问 → 意图识别 → FAQ检索 / 路由Sub-Agent → 调内部接口 → 汇总回复

系统：超时未支付订单 → 定时扫描 → 自动取消 → 回滚库存
```

### 1.4 优先级说明

| 级别 | 含义 |
|------|------|
| **P0** | 必须实现，Demo 演示要求完整可运行 |
| **P1** | 仅提示规则与设计思路，开发时保留接口但可跳过实现 |

## 二、系统架构

### 2.1 服务职责

| 服务 | 核心职责 |
|------|----------|
| **shop-service** | C 端：注册登录、商品浏览搜索、购物车、下单支付、订单管理、物流追踪、售后申请查询。B 端：分类管理、商品 CRUD 与上下架、查看全部订单、售后审核。系统：超时订单自动取消、缓存管理、内部接口 |
| **ai-service** | FAQ 知识库检索、多智能体意图路由与编排（订单查询 / 物流跟踪 / 售后处理）、流式对话（SSE）、反问澄清、兜底回复 |

### 2.2 数据归属

| Schema | 归属服务 | 内容 |
|--------|----------|------|
| `shop` | shop-service | users、categories、products、cart_items、orders、order_items、payment_records、logistics_records、after_sale_requests |
| `customer_service` | ai-service | faq_embeddings（pgvector）、conversations、messages |

### 2.3 基础设施

| 组件 | 用途 |
|------|------|
| **Nginx** | 统一入口，路径路由，CORS |
| **PostgreSQL + pgvector** | 关系数据 + 向量数据统一存储 |
| **Redis** | 热门商品列表、商品详情、分类树缓存 |
| **Docker Compose** | 全部组件容器化，一键启动 |

| 下单 / 确认地址 | C 端 | 确认或填写收货地址、订单金额确认、提交 | shop-service |

#### 2.4.2 交互与体验要求

| 编号 | 要求 | 说明 |
|------|------|------|
| UX-01 | 表单校验 | 注册、登录、下单等表单需前端校验（邮箱格式、密码长度、必填项），提交前拦截明显错误 |
| UX-02 | 加载状态 | 所有异步请求需有加载指示（列表骨架屏、按钮 loading、页面整体加载），避免用户感知卡死 |
| UX-03 | 空状态 | 空列表、无搜索结果、无订单等场景需展示友好的空状态提示，而非空白页面 |
| UX-04 | 错误提示 | 后端返回错误时，以用户可理解的语言展示（如"该邮箱已注册"，而非直接展示错误码），网络异常时提示重试 |
| UX-05 | 操作反馈 | 关键操作（下单、支付、取消、售后申请）成功后需有明确反馈（Toast/弹窗/状态变更），失败时说明原因 |
| UX-06 | 二次确认 | 不可逆操作（取消订单、删除购物车项、删除分类）需弹窗确认，防止误操作 |
| UX-07 | 登录态感知 | 导航栏根据登录状态展示不同内容（未登录：登录/注册按钮；已登录：用户昵称、退出）；Token 过期时自动跳转登录页 |
| UX-08 | 权限感知 | 普通用户看不到管理后台入口；管理员可见 B 端入口；未登录访问需认证页面时跳转登录 |
| UX-09 | 分页与无限滚动 | 商品列表、订单列表等长列表需分页或滚动加载，避免一次性加载全部数据 |
| UX-10 | 缓存感知 | 商品上下架后 C 端应立即反映（下架商品不再出现在列表），若短暂不一致可接受但需在合理时间内（TTL 到期）自动纠正 |

#### 2.4.3 AI 客服对话界面要求

| 编号 | 要求 | 说明 |
|------|------|------|
| AIX-01 | 流式输出 | 对话回复需以打字机效果逐字/逐句呈现，不等待完整回复后一次性展示 |
| AIX-02 | Agent 推理可见 | 对话过程中 SSE 推送的中间推理阶段（路由决策 → Sub-Agent 执行中 → 结果汇总）应以可折叠面板或状态标签形式展示，让用户感知 Agent 的思考过程 |
| AIX-03 | 消息气泡 | 用户消息与 AI 回复以聊天气泡形式区分，支持多轮对话滚动查看历史 |
| AIX-04 | 会话管理 | 支持新建会话、查看历史会话列表、切换会话，会话标题优先使用 AI 自动生成的摘要 |
| AIX-05 | 输入状态 | AI 思考中时展示"正在输入..."动画，输入框暂不可用；完成后恢复 |
| AIX-06 | 错误兜底展示 | AI 调用内部接口失败时，仍以自然语言告知用户"暂时无法查询，请稍后重试"，不做技术性报错展示 |

#### 2.4.4 B 端与 C 端的视觉区分

| 编号 | 要求 | 说明 |
|------|------|------|
| VX-01 | 视觉差异 | B 端管理后台与 C 端用户界面应有明显的视觉区分（如不同的导航配色或布局风格），避免用户混淆 |
| VX-02 | B 端操作效率 | 管理后台以表格、表单、操作按钮为主，强调操作效率而非营销展示 |
| VX-03 | C 端展示导向 | 用户端以商品展示、浏览体验为核心，界面友好、引导清晰 |

#### 2.4.5 跨服务交互要求

| 编号 | 要求 | 说明 |
|------|------|------|
| CX-01 | 统一鉴权 | 前端持同一 JWT 同时访问 shop-service 和 ai-service，用户无需分别登录 |
| CX-02 | 服务切换无感 | 用户在 AI 客服对话中查询订单时，前端不感知 ai-service 调用了 shop-service 内部接口，整个过程对用户透明 |
| CX-03 | 数据一致性 | 用户在 C 端下单后，立即通过 AI 客服查询该订单应能看到最新数据（同一数据库，无延迟问题） |

### 3.1 用户域

**归属服务**：shop-service

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| U-01 | 用户注册 | P0 | 邮箱 + 密码 + 昵称注册，邮箱唯一 |
| U-02 | 用户登录 | P0 | 邮箱 + 密码登录，签发 JWT Token，区分 user/admin 角色 |
| U-03 | 获取个人信息 | P0 | 通过 JWT 获取当前用户信息 |
| U-04 | 更新收货地址 | P0 | 维护收货地址 |
| U-05 | 修改密码 | P0 | 验证旧密码后更新 |
| U-06 | 管理员登录 | P0 | 同登录接口，role=admin 进入管理后台 |
| U-07 | 管理员查看用户列表 | P0 | — |

#### 闭环链路

```
注册 → 登录(签发JWT) → 持Token访问需认证接口 → Token过期(24h)后重新登录

管理员：同登录入口 → JWT含role=admin → 访问B端接口
```

**关键闭环要求**：
- 注册后即可登录，登录后所有需认证接口通过 JWT 识别身份
- 普通用户与管理员的权限隔离在应用层通过 JWT role 实现
- 未登录访问受保护接口应返回 401

### 3.2 商品域

**归属服务**：shop-service（C 端浏览 + B 端管理）

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| P-01 | 分类浏览商品 | P0 | 按分类筛选（含子分类），分页，仅展示 on_sale 商品 |
| P-02 | 关键词搜索商品 | P0 | 名称模糊匹配，仅展示 on_sale 商品 |
| P-03 | 热门商品列表 | P0 | 优先从 Redis 读取，未命中查 DB 回写 |
| P-04 | 商品详情 | P0 | 优先从 Redis 读取 |
| P-05 | 创建分类 | P0 | 支持二级分类树（parent_id） |
| P-06 | 编辑/删除分类 | P0 | 删除前检查是否有商品引用 |
| P-07 | 发布商品 | P0 | 填写名称、描述、价格、图片URL、库存、分类 |
| P-08 | 编辑商品 | P0 | 修改商品字段 |
| P-09 | 商品上下架 | P0 | on_sale ↔ off_sale 切换 |

#### 闭环链路（B 端 → C 端）

```
管理员创建分类 → 发布商品(status=on_sale) → 清除热门商品缓存
    → C端用户浏览/搜索 → 商品列表展示该商品 → 点击查看详情

管理员编辑商品/下架 → 清除对应缓存 → C端即刻看到更新/商品消失
```

**关键闭环要求**：
- B 端操作（发布/编辑/上下架）必须及时反映到 C 端（通过缓存清除实现）
- 下架商品不影响已创建的订单（订单明细是快照）
- 删除分类前必须检查无商品引用，有引用则拒绝

### 3.3 交易域

**归属服务**：shop-service

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| T-01 | 加入购物车 | P0 | 同一商品重复添加时数量叠加（UPSERT） |
| T-02 | 修改购物车数量 | P0 | quantity > 0 |
| T-03 | 删除购物车项 | P0 | — |
| T-04 | 查看购物车 | P0 | JOIN 商品信息，展示合计金额 |
| T-05 | 创建订单 | P0 | 事务内：锁库存 → 扣减 → 生成 pending 订单 → 清空购物车 |
| T-06 | 模拟支付 | P0 | 订单状态 pending → paid，幂等校验 |
| T-07 | 手动取消订单 | P0 | 仅 pending 可取消，事务内回滚库存 |
| T-08 | 订单列表 | P0 | 按时间倒序，支持按状态筛选 |
| T-09 | 订单详情 | P0 | 订单基本信息 + 商品明细快照 |

#### 闭环链路

```
加入购物车 → 查看购物车(确认商品/数量/金额) → 确认收货地址 → 提交订单
    → 事务：锁库存 → 校验库存充足 → 扣减 → 生成pending订单(明细快照) → 清空购物车
    → 用户点击支付 → 校验status=pending → status → paid → 生成支付记录(幂等)
    → 支付完成

[分支-取消] pending订单 → 用户手动取消 → 事务：status → cancelled + 回滚库存
[分支-超时] pending订单超过30分钟 → 系统自动取消 → status → cancelled + 回滚库存（见3.7系统域）
```

**关键闭环要求**：
- 下单、支付、取消均需保证事务原子性
- 支付需幂等——同一订单重复支付应被拒绝
- 库存扣减需加行级锁防止超卖
- 订单明细保存商品名和价格的快照，后续商品变更不影响历史订单
- 仅 pending 状态可支付，仅 pending 状态可取消
- paid 订单不可取消（本期不做退款，退款走售后流程）

### 3.4 物流域

**归属服务**：shop-service

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| L-01 | 查询物流状态 | P0 | 按订单查询，返回当前状态、运输节点时间线、预计送达 |
| L-02 | 物流状态流转（模拟） | P0 | 使用预置演示数据覆盖各状态，管理员可手动推进 |

#### 闭环链路

```
支付成功(订单status=paid) → 系统关联预置物流数据(状态=picked_up)
    → 用户查看物流 → 显示物流状态 + 运输节点 + 预计送达
    → 物流状态模拟流转：picked_up → in_transit → out_for_delivery → delivered
    → 用户看到包裹从揽件到签收的完整过程
```

**关键闭环要求**：
- 支付成功后自动关联预置物流演示数据（不需要对接真实快递 API）
- 物流状态需覆盖四个阶段：已揽件 → 运输中 → 派送中 → 已签收
- 物流状态流转可通过管理员手动推进或预设数据直接展示
- 仅可查询本人订单的物流信息

### 3.5 售后域

**归属服务**：shop-service

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| A-01 | 申请售后 | P0 | 对 paid 订单发起退款(refund)或退货(return)，填写原因 |
| A-02 | 查看售后进度 | P0 | 查看自己的售后申请列表及状态流转 |
| A-03 | 管理员审核售后 | P0 | 审核通过(approved) / 驳回(rejected) / 确认完成(completed) |

#### 闭环链路

```
用户对paid订单发起售后 → 选择类型(refund/return) + 填写原因
    → 创建售后申请(status=pending)
    → 管理员查看 → 审核通过(status=approved) / 驳回(status=rejected)
    → 通过后管理员确认完成(status=completed)
    → 用户随时可查看售后进度
```

**关键闭环要求**：
- 仅 paid 状态订单可发起售后
- 同一订单可发起多次售后申请（不同原因）
- 售后状态流转必须完整：pending → approved/rejected → completed
- P0 阶段必须实现管理员审核功能（这是电商文档 v5.4 中标记为 P1 但需要闭环的关键修正）
- 换货(exchange)类型可保留但实现标记为 P1
- 仅可查询本人的售后申请

### 3.6 AI 客服域

**归属服务**：ai-service（AI 服务为独立项目，shop-service 配合提供内部接口）

#### 3.6.1 多智能体架构

```
用户提问 → Master Agent（意图识别 + 路由）
                ├── 订单查询意图 → OrderAgent → 调用 /internal/orders → 汇总
                ├── 物流查询意图 → LogisticsAgent → 调用 /internal/logistics → 汇总
                ├── 售后查询意图 → AfterSaleAgent → 调用 /internal/after-sales → 汇总
                ├── 商品咨询意图 → 调用 /internal/products → 汇总
                └── 意图不明/超出范围 → FAQ 检索兜底
```

#### 功能点清单

| 编号 | 功能点 | 归属服务 | 优先级 | 说明 |
|------|--------|----------|--------|------|
| AI-01 | FAQ 知识库检索 | ai-service | P0 | 检索 FAQ 问答对，返回匹配答案 |
| AI-02 | 意图识别与路由 | ai-service | P0 | Master Agent 分析用户意图，路由到对应 Sub-Agent 或 FAQ 兜底 |
| AI-03 | 订单查询（Sub-Agent） | ai-service | P0 | 调用 shop-service 内部接口（透传 JWT），获取当前用户订单数据 |
| AI-04 | 物流查询（Sub-Agent） | ai-service | P0 | 调用 shop-service 内部接口（透传 JWT），获取物流数据 |
| AI-05 | 售后查询（Sub-Agent） | ai-service | P0 | 调用 shop-service 内部接口（透传 JWT），获取售后数据 + FAQ 知识库 |
| AI-06 | 反问澄清 | ai-service | P0 | 意图模糊时反问用户，不强行猜测 |
| AI-07 | 流式对话（SSE） | ai-service | P0 | 打字机效果，中间推理过程分阶段推送 |
| AI-08 | 兜底回复 | ai-service | P0 | 超出知识/数据范围时明确告知，不做幻觉回答 |
| AI-09 | 多轮对话 | ai-service | P0 | 会话管理（创建/活跃/关闭），消息记录保持上下文 |
| AI-10 | 用户数据隔离 | ai-service | P0 | 会话绑定 user_id，仅查询当前用户自身数据 |

#### 闭环链路

```
[FAQ 类问题]
用户提问"怎么退货？" → Master Agent 意图识别 → 知识库类 → FAQ 检索 → 返回答案
    → 用户追问 → 多轮对话保持上下文 → 继续回答

[实时查询类问题]
用户提问"我的订单到哪了？" → Master Agent 意图识别 → 订单查询
    → OrderAgent 调用 GET /internal/orders（Header: Authorization: Bearer <用户JWT>）
    → shop-service 解析 JWT → 校验 → 以 user_id 查询 shop.orders → 返回订单数据
    → OrderAgent 返回结果 → Master Agent 汇总为自然语言 → 返回用户

[意图不明]
用户提问"那个东西怎么样了？" → Master Agent 意图模糊
    → 反问用户"您是指订单、物流还是售后？" → 用户澄清 → 继续路由

[超出范围]
用户提问"今天天气怎么样？" → Master Agent 意图识别失败
    → FAQ 兜底 → 未匹配 → "抱歉，我无法回答这个问题"
```

#### shop-service 需暴露的内部接口（供 ai-service 调用）

所有内部接口**复用 JWT 认证**：ai-service 将前端用户的 JWT 放入 `Authorization` 请求头透传给 shop-service，shop-service 解析 JWT 获取 `user_id`，数据隔离由 shop-service 保证。

| 接口 | HTTP 端点 | user_id 来源 | 用途 | 优先级 |
|------|-----------|-------------|------|--------|
| 订单查询 | `GET /internal/orders` | JWT 解析 | 查询当前用户订单列表 | P0 |
| 物流查询 | `GET /internal/logistics?order_id=...` | JWT 解析（并校验订单归属） | 查询指定订单物流状态 | P0 |
| 售后查询 | `GET /internal/after-sales` | JWT 解析 | 查询当前用户售后申请 | P0 |
| 商品查询 | `GET /internal/products/search?keyword=...` | 仅校验 JWT 有效性 | 搜索商品信息 | P0 |

**关键闭环要求**：
- AI 客服不直连数据库，所有业务数据通过 shop-service 内部 HTTP 接口获取
- 内部接口复用用户 JWT 认证：ai-service 透传原始 JWT，shop-service 自行解析获取 user_id 并校验
- 内部接口调用失败或超时时，AI 服务回退到纯 FAQ 回答，不崩溃
- 会话绑定 user_id，用户 A 无法查询到用户 B 的数据（数据隔离）
- 4 个验收场景必须全部走通：①订单查询 ②物流跟踪 ③售后咨询 ④兜底测试
- Agent 执行过程中遇到异常需自我恢复，不中断对话

#### 3.6.2 AI 客服与电商系统的关系

```
ai-service（独立项目）                     shop-service（本项目）
┌──────────────────────┐                ┌──────────────────────┐
│ Master Agent         │                │                      │
│   ├─ OrderAgent      │── HTTP+JWT ──► │ /internal/orders     │
│   ├─ LogisticsAgent  │── HTTP+JWT ──► │ /internal/logistics  │
│   ├─ AfterSaleAgent  │── HTTP+JWT ──► │ /internal/after-sales│
│   └─ FAQ 兜底        │── HTTP+JWT ──► │ /internal/products   │
│                      │                │                      │
│ 直接操作：            │                │  解析JWT→user_id     │
│  customer_service    │                │  校验+查询shop schema │

```

| 编号 | 功能点 | 归属服务 | 优先级 | 说明 |
|------|--------|----------|--------|------|
| S-01 | 超时订单自动取消 | shop-service | P0 | 定时扫描 pending 超过 30 分钟的订单，逐条取消并回滚库存 |
| S-02 | 热门商品缓存 | shop-service | P0 | Cache-Aside 模式，缓存热门列表 + 商品详情 + 分类树 |

| S-05 | 请求追踪 | shop-service | P0 | 每个请求携带 request_id，贯穿所有日志 |
| S-06 | 统一异常处理 | shop-service | P0 | 业务异常统一格式 {code, message, request_id} |
| S-07 | 健康检查 | 全部服务 | P0 | 每个服务暴露 /health 端点 |
| S-08 | 一键部署 | 基础设施 | P0 | Docker Compose 管理全部容器，一键启动 |

#### 闭环链路

```
[超时取消]
APScheduler每5分钟触发 → 查询pending且超过30分钟的订单
    → 无超时订单 → 等待下一轮
    → 有超时订单 → 逐条处理(每条独立事务)：
        锁定订单 → 二次校验仍为pending → status→cancelled → 回滚库存 → 记录日志

[缓存闭环]
C端请求 → 查Redis → 命中返回 / 未命中查DB → 回写Redis(TTL)
B端写操作 → 更新DB → 删除对应Redis key → 下次读触发热加载
Redis不可用 → 降级直接查DB → 不影响业务可用性
```

**关键闭环要求**：
- 超时取消每条订单独立事务，一条失败不影响其他
- 缓存不可用时系统仍可正常运行（降级到 DB）
- 所有服务通过 Docker Compose 一键启动，无需手动配置依赖

## 四、跨系统对接总览

### 4.1 接口调用清单

| 调用方 | 被调用方 | 接口 | 认证方式 | 用途 |
|--------|----------|------|----------|------|
| ai-service | shop-service | `GET /internal/orders` | JWT（透传） | 查询用户订单，user_id 由 shop-service 解析 JWT 获取 |
| ai-service | shop-service | `GET /internal/logistics?order_id=...` | JWT（透传） | 查询物流状态，校验订单归属 |
| ai-service | shop-service | `GET /internal/after-sales` | JWT（透传） | 查询售后进度，user_id 由 shop-service 解析 JWT 获取 |
| ai-service | shop-service | `GET /internal/products/search?keyword=...` | JWT（透传） | 搜索商品，校验 JWT 有效性 |
| 前端 | shop-service | 全部 C 端 / B 端 API | JWT Bearer Token | 用户操作 |
| 前端 | ai-service | AI 对话 API | JWT Bearer Token | 客服对话，ai-service 将 JWT 透传给 shop-service |

### 4.2 数据隔离链路

```
用户A登录 → JWT(user_id=A) → 前端持 JWT 调用 AI 客服
    → ai-service 透传 JWT 调用 /internal/orders
    → shop-service 解析 JWT → user_id=A → 查询 → 返回用户A的订单
    → 用户A无法通过 AI 客服或直接 API 获取用户B的数据（JWT 强制绑定身份）
```

### 4.3 异常降级链路

```
ai-service 调用 shop-service 内部接口
    → 正常返回 → 生成自然语言回答
    → 超时/错误 → 降级为 FAQ 检索 → 如 FAQ 也无匹配 → 友好提示用户稍后重试
    → ai-service 自身不崩溃
```

## 八、附录

### 8.1 关键业务规则速查

| 规则 | 说明 |
|------|------|
| 用户权限隔离 | 普通用户仅操作自身数据；管理员可查看全部订单但不可修改 |
| 订单状态机 | pending → paid / cancelled，不可逆（paid 不退，cancelled 不恢复） |
| 商品状态机 | on_sale ↔ off_sale，下架不影响已有订单（订单明细快照） |
| 物流状态机 | picked_up → in_transit → out_for_delivery → delivered |
| 售后状态机 | pending → approved / rejected → completed |
| 库存并发 | 下单时 FOR UPDATE 行级锁防超卖 |
| 支付幂等 | FOR UPDATE + status 校验 |
| AI 数据获取 | 仅通过内部 HTTP 接口，透传用户 JWT 认证，不直连数据库 |
| 缓存策略 | Cache-Aside：读回写，写删除；不可用时降级 DB |

### 8.2 状态速查

| 实体 | 状态值 | 说明 |
|------|--------|------|
| 订单 | `pending` / `paid` / `cancelled` | 待支付 / 已支付 / 已取消 |
| 商品 | `on_sale` / `off_sale` | 上架 / 下架 |
| 物流 | `picked_up` / `in_transit` / `out_for_delivery` / `delivered` | 已揽件 / 运输中 / 派送中 / 已签收 |
| 售后 | `pending` / `approved` / `rejected` / `completed` | 待处理 / 已通过 / 已拒绝 / 已完成 |
| 会话 | `active` / `closed` | 活跃 / 已关闭 |
